Agentic ETL Specifications

This page outlines the technical specifications for the Agentic ETL system.

What This Covers

  1. What each chat command does
  2. The workflow behind each command (function calls + data passed)
  3. The ideal consumer workflow from Google Drive to OneDrive to local input to AI mapping to config updates
  4. A data-flow diagram with explanations

1) Chat Commands: What Each One Does

These commands are parsed in the frontend chat handler (frontend/assets/js/chat_intake.js) and routed to backend FastAPI endpoints, which call Python modules in backend/src. The backend and frontend communicate via HTTP API calls.

Main workflows

  1. run file processing (shortcut: 1)
    • What it does: Scans files in data/input, detects changed/new files (using hashes and baselines), prepares changed files for analysis, and writes a changed-files manifest. Triggers AI analysis queue if desired.
    • Primary frontend function: runFileProcessing()
    • Backend: backend/src/validate_excel_files.py, backend/src/file_change_detector.py
  2. run file sync (shortcut: 2)
    • What it does: Full sync chain: Google Drive → OneDrive → project input folder. Downloads only new/changed files using eTag and lastModifiedDateTime. Folder structure is not preserved in local input.
    • Primary frontend function: runFileSync()
    • Backend: backend/src/google_to_onedrive_sync.py, backend/src/onedrive_download.py
  3. input (shortcut: 3)
    • What it does: Opens file picker and uploads one CSV/Excel file into data/input. Triggers change detection and shows a summary of changes (if any) before further processing.
    • Primary frontend functions: triggerFileInput() → uploadAndProcess(file)
    • Backend: backend/api/app.py (upload endpoint), backend/src/file_change_detector.py

Tool setup

  1. authenticate onedrive (shortcut: 4)
    • What it does: Starts OneDrive device-code auth (or checks if token already exists).
    • Frontend function: runOneDriveAuthSetup() -> startOneDriveAuth()
  2. complete onedrive auth (shortcut: 5)
    • What it does: Completes the pending OneDrive auth session after browser sign-in.
    • Frontend function: completeOneDriveAuth()

Individual commands for testing

  1. analyze (shortcuts: analyze files, 6)
    • What it does: AI analysis of input file columns against config columns, with change-aware confirmation flow. Uses a local Ollama LLM to generate mapping suggestions and issue flags. Supports both manual upload and server-file auto-detect flows.
    • Frontend entry function: startAnalyze() then analyzeFiles(...) (manual upload flow).
    • Also supports server-file auto-detect flow via analyzeUsingServerFiles(...).
    • Backend: backend/api/llm_prompt.py, backend/api/app.py
  2. download from onedrive (shortcut: 7)
    • What it does: OneDrive-only sync into project input folder. Downloads only new/changed files and updates a manifest log.
    • Frontend function: runOneDriveDownload()
    • Backend: backend/src/onedrive_download.py
  3. run google to onedrive (shortcut: 8)
    • What it does: Legacy alias now routed to full file-sync behavior (Google Drive → OneDrive → input folder).
    • Frontend function: runGoogleToOneDrive()
    • Backend: Alias to run_file_sync()
  4. cleanup (shortcut: 9)
    • What it does: Clears generated state (output, hashes, state, archive, OneDrive cache/manifest) while preserving files in data/input.
    • Frontend function: runCleanup()
    • Backend: backend/api/app.py (cleanup endpoint)

Legacy commands (kept for compatibility)

  1. run excel scripts (shortcut: 10)
    • What it does: Alias to run file processing.
    • Frontend function: runExcelScripts() -> runFileProcessing()
  2. run validate (shortcut: 11)
    • What it does: Validation-only module run.
    • Frontend function: runValidateOnly()
  3. run transform (shortcut: 12)
    • What it does: Transform-only module run.
    • Frontend function: runTransformOnly()

Support command: help or 0 or ? shows the command menu.

2) Workflow of Each Command (Function Calls + Data Passed)

The backend is a FastAPI app (backend/api/app.py) exposing endpoints for each command. File processing and change detection are handled by backend/src/validate_excel_files.py and backend/src/file_change_detector.py. AI analysis is performed by backend/api/llm_prompt.py, which calls a local Ollama LLM. File sync is handled by backend/src/google_to_onedrive_sync.py and backend/src/onedrive_download.py. Config update automation is scaffolded in backend/src/prompt_auto_handle.py but not yet fully implemented.

A. run file processing

  1. Chat input is normalized in sendUserMessage().
  2. runFileProcessing() calls fetch('http://localhost:5001/run/file-processing', { method: 'POST' }).
  3. Backend run_file_processing() runs module backend.src.validate_excel_files.
  4. validate_excel_files calls:
    • list_excel_files(input_dir)
    • compare_file_without_updating(file) for each file
    • run_transform(...) for changed/new files
    • writes changed-file list to data/state/.changed_files.json
  5. API response includes status, message, processing_counts, file_details.changed_files, subprocess stdout and stderr.
  6. Frontend can trigger guided per-file analysis queue from file_details.changed_files.

Data passed: Request body none (default path); response includes changed file names and run logs.

B. run file sync

  1. runFileSync() uses callApi('/run/file-sync', ...).
  2. Backend run_file_sync() executes:
    • run_module(['backend.src.google_to_onedrive_sync'])
    • if success, then run_module(['backend.src.onedrive_download'])
  3. Returns combined status + per-stage output and parsed sync counts.

Data passed: Request body none; response returns status, message, and stage payloads with returncode, stdout, stderr, and sync_counts.

C. input

  1. triggerFileInput() opens browser file picker.
  2. On file change, frontend runs uploadAndProcess(file).
  3. Upload call uses POST /upload_and_process with FormData key file.
  4. Backend saves file into data/input/<file_name> and runs compare_file(file_name).
  5. Frontend then calls runAutoChangeDetectionForUpload(file.name) using POST /check_file_changes and JSON body { "file_name": "...", "update_baseline": false }.
  6. User is prompted to run run file processing when ready.

Data passed: Upload request is multipart form file; change-check request is JSON file name plus baseline flag.

D. authenticate onedrive

  1. runOneDriveAuthSetup() -> startOneDriveAuth().
  2. Frontend calls POST /run/onedrive_auth/start.
  3. Backend loads or creates MSAL token cache and starts device flow.
  4. Response status:
    • ok and already_authenticated=true if token is already valid
    • pending with auth_id, user_code, and verification URI fields if user must sign in

E. complete onedrive auth

  1. Frontend sends stored oneDriveAuthSessionId.
  2. Call uses POST /run/onedrive_auth/complete with JSON { "auth_id": "<session_id>" }.
  3. Backend checks pending device flow and returns pending, ok, or error.

F. analyze

There are two supported analyze workflows.

F1) Manual two-file upload flow (/analyze-excel_files)

  1. startAnalyze() initializes analyzeState and asks for INPUT file.
  2. After input file selection, user picks CONFIG file (or saved config is reused).
  3. analyzeFiles(inputFile, configFile) runs a two-phase API flow.

Phase 1 (prepare only): sends input and config files, confirm flags, and retry settings; backend builds change-aware prompt context and may return confirmation-needed state.

Phase 2 (execute prompt): sends prepared prompt fields; backend calls query_llm(prepared_prompt) and returns ai_summary + context and timing metadata.

F2) Auto-detect server-file flow (/analyze-excel_files-from-input)

  1. Frontend call: analyzeUsingServerFiles(...) -> analyzePreparedInputFile(null, options).
  2. Request uses JSON fields such as input_file_name, config_file_name, save_default_config, confirm_ai_analysis, and prepare_only.
  3. Backend selects latest usable input in data/input and config in data/config.
  4. Depending on config status it can return needs_config_upload, single, or needs_config_selection.
  5. For changed or no_change statuses it returns needs_analysis_confirmation unless confirmed.
  6. On confirmed run it builds prompt and calls LLM.

G. download from onedrive

  1. Frontend runOneDriveDownload() -> callApi('/run/onedrive_download', ...).
  2. Backend runs python -m backend.src.onedrive_download.
  3. OneDrive module acquires token, lists folder via Graph API, compares with manifest, downloads changed files into data/input, and updates data/state/onedrive_manifest.xlsx.

H. run google to onedrive

  1. Frontend sends POST /run/google_to_onedrive.
  2. Backend route is an alias to run_file_sync().
  3. Operationally same behavior as run file sync.

I. cleanup

  1. Frontend sends POST /run/cleanup.
  2. Backend clears generated directories:
    • data/output
    • data/state/hashes
    • data/state
    • data/archive
    • OneDrive local cache folder
    • OneDrive manifest file
  3. Preserves data/input.

J. Legacy commands

run excel scripts: Calls runFileProcessing(); same request/data model as command A.

run validate: Calls POST /run/validate; runs only backend.src.validate_excel_files.

run transform: Calls POST /run/transform; runs only backend.src.transforming_excel.

3) Consumer Workflow (Ideal End-to-End)

All state, hashes, and responses are stored in data/state, data/archive, and data/responses. The frontend chat interface (see frontend/assets/js/chat_intake.js) handles file uploads, command parsing, and displays results. It interacts with the backend via fetch calls to endpoints like /run/file-processing, /upload_and_process, etc. The frontend supports both manual and auto-detect flows for analysis.

This is the target operating flow for consumers.

  1. Sync source data from Google Drive into OneDrive
    • Run run file sync.
    • Stage 1 updates OneDrive backup folder from Google Drive.
  2. Sync OneDrive files into local project input
    • Stage 2 of run file sync (or download from onedrive) downloads changed files into data/input.
  3. Detect what actually changed and prepare files
    • Run run file processing.
    • System compares each input file against saved baselines and marks only changed/new files for analysis.
    • Changed-file manifest is written to data/state/.changed_files.json.
  4. Run file processing to generate and save LLM responses
    • Run run file processing. For each changed input file, the system uses the AI model to generate mapping suggestions and issue flags, saving each response as a record in data/responses/.
    • Responses are not immediately applied to config files; they are saved for review and approval.
  5. Review and apply LLM responses to update config
    • Run process responses. This command allows you to review each saved LLM response, preview the proposed mapping/config changes, and approve or skip each one.
    • Approved changes are (or will be) applied to config files in data/config via a deterministic Python update script (automation in progress).
  6. Update config using Python based on approved mappings
    • Ideal state: parse AI mapping output into structured records, then apply those mappings to config files in data/config via a deterministic Python update script.
    • Recommended controls:
      • Keep a backup/version of old config before overwrite.
      • Validate mapped columns exist in input.
      • Run a dry-run preview before final write.
  7. Re-run processing with updated config and verify output
    • Re-run run file processing and/or analyze to validate the new mapping behavior.

Important current-state note: Today, the code clearly generates AI mapping suggestions (ai_summary) and saves LLM response records in data/responses/. However, there is not yet a dedicated built-in step that automatically writes those suggestions back into config schema files. The config-update automation is scaffolded in backend/src/prompt_auto_handle.py and will be implemented as a separate Python action/script for production reliability. Human review is required before config changes are applied.

4) Data Flow Diagram + Explanations

The project uses Docker Compose with three main services: web (frontend), etl (processing), and api (backend). Volumes are mounted for persistent data and code sharing. All major logic is in backend/src and frontend/assets/js.

flowchart TD A[Google Drive Source Files] --> B[run file sync\nPOST /run/file-sync] B --> C[backend.src.google_to_onedrive_sync\nGoogle -> OneDrive backup] C --> D[OneDrive Remote Folder] D --> E[backend.src.onedrive_download\nGraph API + manifest compare] E --> F[data/input\nlocal project input files] F --> G[run file processing\nPOST /run/file-processing] G --> H[backend.src.validate_excel_files\nchange detection + prepare changed files] H --> I[data/state/.changed_files.json] H --> J[data/archive\nbaselines/history] F --> K[analyze\nPOST /analyze-excel_files or /analyze-excel_files-from-input] L[data/config\nselected config file] --> K K --> M[extract_columns + prompt build] M --> N[query_llm via Ollama] N --> O[ai_summary\nmapping suggestions] O --> P[Human review or automation parser] P --> Q[Python config update step\nwrite updated config in data/config] Q --> K

If the diagram does not render, check internet access for the Mermaid CDN script and refresh the page.

Diagram explanation by stage

  1. Source sync stage: Google Drive source files are synced into OneDrive through the file-sync command.
  2. Download stage: OneDrive downloader only fetches new/modified files, then writes them to data/input.
  3. Change-detection stage: Processing reads data/input, compares with saved baselines in archive/hash state, and records changed files.
  4. Analysis stage: Analyze combines selected input and config, extracts column structure, creates a change-aware prompt, and gets LLM mapping suggestions.
  5. Config-update stage: Approved mapping suggestions are applied to config using Python logic, then saved back into data/config.
  6. Closed-loop iteration: Updated config feeds the next analysis run, enabling iterative improvement.
